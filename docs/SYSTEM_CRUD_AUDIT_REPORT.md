# 系统 CRUD 审计报告

> 审计时间：`2026-04-28`
> 审计口径：以当前代码中的页面路由、API 路由和 ORM 写入路径为准

## 1. 结论先行

当前系统的 CRUD 完整性比旧文档描述的更高，尤其是以下三点已经补齐：

- 分析记录：已有完整 `Create / Read / Update / Delete`
- 回测结果：回测完成后可自动持久化，并支持列表查看与删除
- 自选股：已有新增、列表、编辑备注/排序、删除

目前最明显的“未完全闭环”不再是历史上那几项，而是：

- 用户无法自助注销账号
- Legacy 聊天记录仍然只有写入和查看，没有独立清理入口
- 新闻、行情、实时监控等本质上属于只读查询模型

## 3. 关键事实

### 3.1 分析记录已经不是缺口

当前已有：

- `POST /auth/profile/records/analysis`
- `GET /api/user/records/analysis`
- `PUT /api/user/records/analysis/<id>`
- `DELETE /api/user/records/analysis/<id>`

旧文档中“分析记录不完整”的结论已不再成立。

### 3.2 回测结果已经有持久化

当前回测流程里：

- `POST /api/analysis/backtest` 执行策略回测
- 登录用户会触发 `_save_backtest_result(...)`
- `GET /api/user/backtests` 查看历史
- `DELETE /api/user/backtests/<id>` 删除历史

因此，回测并不是一次性前端展示结果。

### 3.3 管理员用户列表已经有分页

`/admin/users` 当前支持：

- `page`
- `per_page`
- `keyword`

所以“后台用户列表无分页”的旧结论也已经失效。

## 4. 仍然存在的真实问题

### 4.1 用户自助注销缺失

普通用户当前没有“删除我的账号”入口，只能通过管理员删除。

### 4.2 AI 新旧记录双轨并存

AI 助手主链路已经使用：

- `user_ai_conversation`
- `user_ai_message`

但成功回复后仍可能镜像写入：

- `user_chat_history`

这让数据模型同时承担“新体系”和“兼容展示”两种职责。

### 4.3 回测实现有双入口概念

- 当前正在运行的回测器：`app/api/analysis_api.py` 内部 `BacktestEngine`
- 高级预留版：`app/services/backtest_engine.py`

从 CRUD 角度看不影响功能，但会影响后续维护的一致性。

## 5. 推荐动作

1. 为普通用户补一个受保护的自助注销流程，至少支持软删除。
2. 为聊天历史明确迁移路线，避免长期双写。
3. 把当前生效的回测实现提炼到 service 层，统一调用点。
4. 如果后续要继续做审计，建议引入接口级测试而不是仅靠文档检查。
